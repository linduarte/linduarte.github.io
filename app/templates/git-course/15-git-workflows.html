Git Workflow para iniciantes

O controle de versão com Git é uma habilidade essencial para qualquer desenvolvedor. Conforme os projetos aumentam em complexidade, não usar Git leva à desorganização, queda de produtividade e erros evitáveis.

O que veremos agora é como se fosse um reforço de tudo que acabamos de estudar mas agora com a visão de fluxo de trabalho, 'workflow', seja como desenvoldevor solo ou em equipe.

Como o Git habilita metodologias ágeis

Desenvolver software é desafiador, especialmente com equipes grandes e distribuídas. Tarefas como acompanhar requisitos, organizar problemas, revisar mudanças de código e lançar software exigem uma coordenação segura.

O Git habilita diretamente metodologias ágeis de software que resolvem muitos desses desafios organizacionais. Por exemplo, o Git dá suporte a:

  • Integração e entrega contínuas (CI/CD) - Os desenvolvedores podem mesclar código várias vezes ao dia para detectar bugs de integração antecipadamente e lançar com frequência.
  • Sprints curtos e iterativos - Problemas e funcionalidades são acompanhados em pequenos lotes que podem ser desenvolvidos e entregues em sprints de 1-2 semanas.
  • Prototipagem rápida - Com ramificação e mesclagem acessíveis, os desenvolvedores podem criar protótipos de recursos experimentais sem medo de quebrar o código de trabalho.

De acordo com a pesquisa de 2022 do Stack Overflow, quase 90% dos desenvolvedores agora usam o Git, embora ele só tenha sido amplamente adotado na última década.

Conceitos básicos no Git

Antes de mergulhar nos fluxos de trabalho, vamos cobrir alguns conceitos-chave do Git:

  • Repository (Repo):Uma pasta que o Git rastreia. Ele contém os arquivos do seu projeto e uma pasta .git oculta com todo o histórico de versões.
  • Commit:Um instantâneo do seu projeto em um determinado momento. Cada commit tem um ID exclusivo e uma mensagem descrevendo as alterações.
  • Branch:Uma linha separada de desenvolvimento. Pense nisso como um universo paralelo para o seu projeto.
  • Merge:Combinando mudanças de um ramo para outro.
  • Remote:Uma versão do seu repositório hospedada online (por exemplo, no GitHub ou GitLab).

Fluxos de trabalho do Git (Git workflows)

Um fluxo de trabalho do Git define como você usa o Git para gerenciar seu projeto. Aqui estão alguns fluxos de trabalho populares, explicados de forma simples:

  • Fluxo de trabalho básico (para desenvolvedores solo)

Este fluxo de trabalho é simples e perfeito se você estiver trabalhando sozinho.

  • Inicializar um repositório Git:comece a acompanhar seu projeto com o Git. 
git init
  • Mudanças de "stage":Marque os arquivos que você quer incluir no seu próximo commit. Você pode usar . (git add .) para adicionar todos os arquivos. 
git add <file name<
  • "Comite" alterações:salve um instantâneo de suas alterações.Confirmar alterações: salve um instantâneo de suas alterações.
git commit -m "Add a meaningful commit message"
  • Enviar para remoto:carregue seu trabalho em um repositório remoto como o GitHub.
git push origin main
  • Fluxo de trabalho centralizado (para equipes ou desenvolvedores individuais)

Esse é o fluxo de trabalho mais simples e funciona melhor para equipes pequenas. Todo mundo trabalha na mesma branch (geralmente main ou master) e faz os commits das mudanças direto.

  • Puxar as últimas alterações
git pull
  • Faça alterações:faça suas alterações, como adicionar novos recursos, corrigir bugs, etc., preparar e confirmar alterações.
git add .
git commit -m "commit message"
  • Envie as mudanças:faça suas alterações como adicionar novas funcionalidades, corrigir bugs, etc., prepare e registre as mudanças.
git push

Prós: Fácil de entender. Sem necessidade de várias ramificações.

Contras: Pode levar a conflitos se várias pessoas trabalharem nos mesmos arquivos.

Casos de uso: equipes pequenas ou projetos onde os desenvolvedores confiam uns nos outros para não sobrescrever código crítico.

  • Fluxo de Trabalho com Branch de Funcionalidade (Para Times ou Desenvolvedores Solo)

Esse fluxo de trabalho permite que você isole novos recursos ou experimentos do código principal. Esse fluxo de trabalho é ótimo para equipes. Cada nova funcionalidade ou correção recebe seu próprio branch, mantendo o principal estável. Para começar, você precisa clonar o repositório de código.

  • Clonar repositório:clone o repositório principal no qual você ramificará seu recurso
git clone <repo-url>
  • Crie um novo branch:Crie um novo branch para sua funcionalidade e comece a trabalhar na funcionalidade sem afetar o branch principal.
git checkout -b feature/my-new-feature
  • Faça alterações e commite:trabalhe na sua funcionalidade e salve seu progresso.
 git commit -m "Implement feature X"
  • Envie para o remoto:compartilhe seu branch principal atualizado.
 git push -u origin origin feature/my-new-feature

Crie uma PR (solicitação de pull) para mesclar seu branch no main.

Observação: Uma vez que a funcionalidade esteja completa, o branch pode ser mesclado ao branch principal (ou de desenvolvimento), muitas vezes através de um pull request ou merge request.

Prós: Mantém o ramo principal limpo. Fácil revisar as mudanças antes de fazer a mesclagem.

Contras: Um pouco mais complexo.

Casos de uso: Equipes trabalhando em várias funcionalidades ao mesmo tempo. Isso promove o desenvolvimento isolado.

  • Fluxo de Trabalho Forking (Para Projetos de Código Aberto)

Esse fluxo de trabalho é comum para contribuir com projetos de código aberto hospedados em plataformas como o GitHub.

  • Bifurcar o repositório:faça sua própria cópia do projeto clicando no botão "Bifurcar" no GitHub.
  • Clone o Fork:Baixe seu fork para o seu computador.
  git clone <fork-repo-url>
  • Crie uma Branch de Funcionalidade:Trabalhe nas suas mudanças em uma branch separada.
 git checkout -b feature/fix-bug
  • Envie para o seu Fork:Carregue seu branch no seu repositório forkado.
 git push origin feature/fix-bug
  • Envie um Pull Request (PR):Proponha suas mudanças para o repositório original.
